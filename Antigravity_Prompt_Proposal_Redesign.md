## 1. Design system (apply as Tailwind tokens / CSS variables)

**Typography**
- Display/headings: Plus Jakarta Sans (weights 600–800)
- Body/data: Inter (weights 400–600)

**Layout**
- Max content width ~920px, centered, generous section padding (32–44px).

- Section titles: bold, with a small 4px-wide gold vertical bar to the left (`::before` pseudo-element), not numbered markers — this isn't a sequential process, so don't add 01/02/03 step numbering to section titles (only to line items, where numbering is meaningful).

## 2. Section-by-section structure

**A. Header (dark navy card, rounded bottom corners)**
- Left: Meaven logo mark + wordmark + "GLASS & ALUMINIUM EXECUTION" sub-label.
- Right: "PROPOSAL" tag, proposal number + issue date (from `proposal.id` / `proposal.date`), a "VALID X DAYS" badge computed from the proposal's validity field.
- Below: subject/project reference line (from `proposal.subject` field), then a facts row — Billed To, Date Issued, Client GSTIN — pulled from the client record.

**B. Executive Summary**
- Single card, plain paragraph. This text should be **AI-generated by Gemini** based on the scope of items in the proposal (see Section 4 below) — do not hardcode boilerplate.

**C. Spec Comparison ("Where We've Upgraded Your Spec")**
- Only render this section if the form has both a "client specified" and "our proposed" spec entered for at least one line item — it should not appear on proposals with no BOQ comparison.
- Per item: two-column layout (client-specified vs Meaven-proposed), plus a one-line "Upgrade:" callout underneath summarizing the delta in plain terms.
- The two spec columns pull directly from form fields (`item.client_spec`, `item.proposed_spec`). **The "Upgrade:" callout line is the one part that should be AI-generated** (Gemini), summarizing the practical difference between the two specs in one sentence — see Section 4.

**D. Commercial Specifications (line items)**
- One card per line item, numbered (01, 02, 03…) since this is a real sequence tied to the pricing table below it.
- Each card shows: item name, spec tags (chips — pull from `item.tags[]`), description/hardware notes (from `item.description`), a dimension chip if applicable (`item.dimensions`), and on the right: quantity + unit, rate, and line amount.
- **Item-level discount**: add an editable "Item discount %" input per line item (bind to `item.discount_percent`, default 0). When >0, show the original rate struck through next to the discounted rate. Recalculate that line's amount live.
- Fix from the current app: qty/rate/amount must stay in a fixed-width right-aligned column and never overlap or bleed into each other regardless of number length — this was a known bug in the old template, so please explicitly test with 4+ digit quantities and rates.

**E. Totals**
- Subtotal (sum of line amounts after item-level discounts)
- **Quote-level discount**: editable % field (bind to `proposal.discount_percent`, default 0), shown as a negative line
- Taxable Value = Subtotal − Quote-level discount

- Grand Total (dark navy bar, large type)
- Amount in words, auto-generated from the Grand Total (Indian numbering — Lakh/Crore, not Million/Billion)

**F. Project Execution Timeline**
- Horizontal step cards, pulled from `proposal.timeline[]` (day range + title). Not hardcoded to 4 steps — render however many the form has.

**G. Terms of Partnership (payment milestones)**
- Card per milestone, pulled from `proposal.payment_milestones[]` (label + percentage). Amount per milestone = percentage × Grand Total, recalculated live whenever discounts change upstream.

**H. Care Notes & Terms and Conditions**

## 3. Discount calculation logic (implement as a shared utility, not per-component math)

```
line_amount = rate * (1 - item.discount_percent/100) * qty
subtotal = sum(line_amount for all items)
quote_discount_amount = subtotal * (proposal.discount_percent/100)
taxable_value = subtotal - quote_discount_amount
gst_amount = taxable_value * (gst_rate/100)
grand_total = taxable_value + gst_amount
milestone_amount[i] = grand_total * (milestone[i].percent/100)
```
All of the above must recalculate live as any discount field is edited, including on the client-facing proposal view if discounts are editable there, and must feed the PDF export so the exported PDF always matches what's on screen.

## 4. Where AI (Gemini 2.5 Flash) should generate content

Only these two things should be AI-generated per proposal — everything else pulls from form data:
1. **Executive Summary paragraph** — 2–4 sentences, professional tone, summarizing the scope of items in the proposal and the general value proposition (durability/finish upgrade, certified fabrication network, etc.) without inventing pricing or spec details not present in the item data.
2. **The "Upgrade:" one-line callout** under each spec comparison row — a plain-language summary of the practical difference between the client-specified spec and Meaven's proposed spec for that item, generated from `item.client_spec` and `item.proposed_spec`.

Everything else (specs, tags, dimensions, pricing, notes, T&C, bank details) must come directly from form fields or settings — never AI-generated or hardcoded in the template.
